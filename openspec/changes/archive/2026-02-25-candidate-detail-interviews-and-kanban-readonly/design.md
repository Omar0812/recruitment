## Context

当前系统中，面试记录的查看入口只存在于岗位看板卡片的展开面板。候选人详情页的"投递记录" tab 只展示岗位/阶段/状态，无法在候选人视角查看面试历史。同时，看板中已淘汰候选人与进行中的混排，且仍渲染完整操作按钮，存在误操作风险。

后端已有 `GET /api/interviews?link_id=xxx` 接口（app/routes/interviews.py），无需新增 API。

## Goals / Non-Goals

**Goals:**
- 候选人详情"投递记录" tab 每条记录可展开查看面试记录列表
- 候选人详情 header 对已淘汰候选人展示最近流程信息（岗位名 + 淘汰原因）
- 看板卡片区分 active / inactive 状态，inactive 只读灰化
- 看板每列默认折叠已淘汰卡片，提供 toggle 显示

**Non-Goals:**
- 不在候选人详情页新增面试记录（新增入口保留在看板）
- 不修改面试记录的数据结构或 API
- 不改动流程跟进页（C/D 方向留待后续）

## Decisions

### 1. 面试记录懒加载 vs 随投递记录一起加载

选择**懒加载**：用户点击展开某条投递记录时，才调用 `GET /api/interviews?link_id=xxx`。

理由：候选人可能有多条投递记录，一次性加载所有面试记录会增加首屏请求量；懒加载与看板现有的 `loadInterviewList(linkId)` 模式一致，复用逻辑。

### 2. 已淘汰卡片的渲染方式

选择**同列折叠**而非单独分区：已淘汰卡片折叠在各自所在列的底部，用"显示已淘汰 (N)"toggle 展开，而不是移到单独的"已淘汰"列。

理由：已淘汰候选人在哪个阶段被淘汰是有意义的信息（体现漏斗），保留在原列更直观；单独列会破坏看板的阶段语义。

### 3. inactive 卡片的操作按钮处理

完全不渲染操作按钮（而非渲染后禁用）。理由：禁用按钮仍占空间且视觉噪音大；不渲染更干净，且 `renderCard` 已接收 `lnk` 对象，可直接判断 `lnk.outcome`。

## Risks / Trade-offs

- [风险] 看板折叠 toggle 的状态在 `renderPipeline` 重新渲染后会重置 → 可接受，看板本身每次操作后都全量重渲染，这是现有模式
- [风险] 候选人详情展开面试记录后，如果在看板新增了面试记录，详情页不会自动刷新 → 可接受，用户可手动刷新；两个视图的同步不在本次范围内
