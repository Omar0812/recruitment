## Context

流程跟进页（`renderPipelineTracking`）当前是一张平铺大表，所有活跃候选人混排，无分组无优先级。HR 有三种使用场景：查某个候选人进展、看某个岗位整体进展、日常扫视推进。平铺表格对后两种场景体验差。

看板 header（`renderPipeline`）有"编辑岗位"按钮，语义错误——用户在看候选人管道时不应该看到岗位编辑入口，该入口应在岗位库列表。

## Goals / Non-Goals

**Goals:**
- 流程跟进页改为分组卡片视图，默认按岗位分组，支持切换按阶段分组
- 保留搜索（跨分组实时过滤）
- 看板 header 移除"编辑岗位"按钮
- 岗位库列表操作列加"编辑"链接

**Non-Goals:**
- 不做"今日待跟进"智能提醒（留待后续迭代）
- 不改动后端 API
- 不改动看板卡片的其他交互

## Decisions

### 1. 分组卡片 vs 分组表格

选择**分组卡片**：每个岗位/阶段是一个独立 card，内部用紧凑表格列出候选人。

理由：卡片有视觉边界，一眼能看出有几个岗位在跑、每个岗位几个人；平铺表格加分组 header 视觉层次弱，容易看花。

### 2. 分组切换的实现

用前端切换（不重新请求 API）：数据一次性加载，切换分组维度只是重新渲染。与现有 `renderContent` 模式一致。

### 3. 看板"编辑岗位"按钮的去向

移至岗位库列表每行操作列，加"编辑"链接（`href="#/jobs/edit/${j.id}"`）。看板 header 只保留"← 返回"和岗位标题。

## Risks / Trade-offs

- [风险] 分组卡片在候选人很多时（单岗位 20+ 人）卡片会很长 → 可接受，后续可加卡片内折叠；当前数据量不大
- [风险] 搜索跨分组过滤后，空分组是否显示 → 设计决策：空分组隐藏，保持视图干净
