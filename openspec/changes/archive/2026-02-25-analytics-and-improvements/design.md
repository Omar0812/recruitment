## Context

系统当前无数据分析视图，候选人可同时存在多个活跃流程，多处 P0/P1 Bug 影响数据可靠性，表单无输入验证。本次改动涉及前端新增页面、后端逻辑修复，无新增外部依赖，无数据库结构变更。

## Goals / Non-Goals

**Goals:**
- 新增数据分析页面，纯前端聚合现有 API 数据
- 候选人单活跃流程限制（后端拦截 + 前端提示）
- 修复软删除候选人详情页可访问问题
- 修复编辑候选人不写历史记录
- 前端 API 统一错误处理 + 防重复提交
- 表单输入验证（手机/邮箱/年龄/工作年限）

**Non-Goals:**
- 不做服务端图表渲染（纯 CSS/HTML 实现，不引入 Chart.js 等库）
- 不做数据导出
- 不做权限管理
- 不做历史数据迁移

## Decisions

### 决策 1：数据分析页面纯前端聚合，不新增 API
- **选择**：复用 `/api/pipeline/active`、`/api/jobs`、`/api/candidates` 现有接口，前端 JS 做聚合计算
- **理由**：无需改动后端，数据量小（内部工具），实现最快
- **替代方案**：新增 `/api/analytics` 聚合接口 → 改动更大，当前数据量不需要

### 决策 2：图表用纯 CSS 进度条实现，不引入第三方库
- **选择**：漏斗用百分比宽度的 `<div>` 色块，分布用横向进度条
- **理由**：项目无构建工具，引入 Chart.js 需要 CDN，增加外部依赖风险
- **替代方案**：引入 Chart.js CDN → 增加外部依赖，离线不可用

### 决策 3：单活跃流程在后端和前端双重拦截
- **后端**：`POST /api/pipeline/link` 检查候选人是否已有任意活跃 link（不限岗位），若有返回 400 + 提示当前所在岗位名
- **前端**：捕获 400 错误，显示具体提示信息（"该候选人已在 XX 岗位流程中，请先结束再投递"）
- **理由**：后端保证数据完整性，前端提供友好提示

### 决策 4：API 错误处理用 toast 提示，不用 alert
- **选择**：在 `app.js` 顶部实现轻量 `showToast(msg, type)` 函数，用绝对定位 div 显示 2 秒后消失
- **理由**：`alert()` 阻塞 UI，体验差；引入 toast 库过重
- **替代方案**：继续用 `alert()` → 体验差，阻塞操作

### 决策 5：防重复提交用按钮 disabled 方式
- **选择**：封装 `withLoading(btn, asyncFn)` 工具函数，执行期间 disable 按钮并改文字为"保存中..."
- **理由**：统一处理，不需要每个按钮单独写逻辑

## Risks / Trade-offs

- [风险] 数据分析页面数据量大时前端聚合慢 → 当前是内部小工具，候选人数量有限，可接受
- [风险] 单活跃流程限制会影响"转移岗位"流程 → 转移岗位先 withdrawn 再 link，withdrawn 后不再是活跃 link，不受影响
- [风险] toast 提示与现有 alert 混用 → 逐步替换，新增错误处理用 toast，现有 alert 暂不动
