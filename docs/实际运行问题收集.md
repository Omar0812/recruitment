### #1 多文件批量建档时，简历解析又慢又脆弱

**「简单说」**

- 什么问题：一次拖入多份简历（比如 50 份）建档时，系统会一个接一个地上传、一个接一个地让 AI 解析，整个过程非常慢
- 什么时候会遇到：批量导入简历的时候，文件越多越明显
- 会怎样：上传阶段没有进度提示，干等着；解析阶段每份要 3-15 秒，50 份就是好几分钟；中间任何一份出错，后面的全部停掉不解析了
- 修好后：改成"点一道做一道上一道"的模式——你点开第 1 个人的表单时才让 AI 读第 1 份简历，点下一个人时才读第 2 份。不再一口气把 50 道菜全做好再上桌，也不会因为其中一道菜出问题导致整桌停工

**「技术细节」**

- 优先级：P2
- 位置：`frontend/src/composables/useCandidateCreate.ts` → `confirmGroups()`（约第 254-295 行）
- 现象：`confirmGroups()` 先串行 `await` 逐个上传全部文件（第 262-266 行），再串行 `await` 逐个调用 AI 解析（第 285-289 行）。用户已进入第 1 个表单编辑，后台仍在解析第 2-50 个文件
- 问题：
  1. 上传阶段串行逐个 `await`，无并发，无进度反馈
  2. 解析阶段串行逐个 `await`，每个 3-15 秒，50 个 = 数分钟，`state.parsing` 反复闪烁
  3. 解析中间任一个抛异常，整个 for 循环中断，后续文件全部不解析
  4. 升级 vision API 后单次解析更慢，问题会加剧
- 方案：改为"点开谁才解析谁"——用户点进某个候选人的表单时才触发该人的 AI 解析，不在 `confirmGroups` 中批量解析。上传阶段改为并发（`Promise.all` 或限流并发），加进度提示。每个人的解析独立，一个出错不影响其他人

### #2 面评和背调的结论显示英文，HR 看不懂

**「简单说」**

- 什么问题：候选人时间线上，面试评价显示 "pass" 或 "reject"，背景调查显示 "pass" 或 "fail"，都是英文
- 什么时候会遇到：在「进行中」页面展开任何候选人的事件时间线，只要有面评或背调记录就会看到
- 会怎样：HR 不一定看得懂这些英文，容易误解
- 修好后：显示中文——面评：「通过」/「淘汰」；背调：「通过」/「未通过」

**「技术细节」**

- 优先级：P0
- 位置：`frontend/src/components/pipeline/EventCard.vue` → `conclusion` computed（约第 149 行）
- 原因：`conclusion` computed 对 `interview_feedback` 类型直接返回 `payload.conclusion`（值为 `"pass"` / `"reject"`），对 `background_check_result` 类型直接返回 `payload.result`（值为 `"pass"` / `"fail"`），没有做中文映射
- 方案：在 `conclusion` computed 里加映射：
  - `interview_feedback`：`pass` → `通过`，`reject` → `淘汰`
  - `background_check_result`：`pass` → `通过`，`fail` → `未通过`
- 影响面：进行中页面所有候选人的事件时间线展示

### #3 关闭岗位时，被淘汰的候选人状态值写错了 ✅ 已修复

**「简单说」**

- 什么问题：关闭一个岗位时，系统会自动淘汰该岗位下所有还在流程中的候选人，但淘汰原因被写成了中文"岗位关闭"，而系统只认英文枚举值
- 什么时候会遇到：关闭一个还有候选人在流程中的岗位
- 会怎样：数据分析统计时，这些候选人不会被归入"已淘汰"类别，而是变成一个孤立的"岗位关闭"分类；前端筛选也可能识别不了这个值；数据库里的数据不干净
- 修好后：状态值正确写为 `rejected`，"岗位关闭"这个原因记录在事件详情里，统计和筛选都正常

**「技术细节」**

- 优先级：P0
- 位置：`app/entry/api/jobs.py` → `close_job` 函数（约第 161-163 行）
- 原因：代码直接写了 `application.outcome = "岗位关闭"`，而 `Outcome` 枚举（`app/models/enums.py`）的合法值只有 `hired` / `left` / `rejected` / `withdrawn`
- 方案：
  - `application.outcome` 改为 `Outcome.REJECTED.value`（即 `"rejected"`）
  - 把"岗位关闭"信息放到 Event 的 payload 里，如 `payload={"outcome": "rejected", "reason": "岗位关闭"}`
- 影响面：岗位关闭功能 + 数据分析 + 候选人流程记录

### #4 关闭岗位淘汰候选人时，没有留下任何记录 ✅ 已修复

**「简单说」**

- 什么问题：接上条，关闭岗位自动淘汰候选人时，只改了状态，但没有在候选人的时间线上写一条"因岗位关闭被淘汰"的记录
- 什么时候会遇到：关闭一个有多个活跃候选人的岗位后，去看这些候选人的时间线
- 会怎样：候选人的时间线上看不到是什么时候、因为什么被淘汰的，记录断裂；候选人的阶段（如"面试中"）不会更新，因为阶段是根据事件推算的；简报里可能还会显示这些已经被淘汰的候选人的待办
- 修好后：每个被淘汰的候选人时间线上都有一条"岗位关闭"的结束记录，阶段正确更新，简报数据准确

**「技术细节」**

- 优先级：P0
- 位置：`app/entry/api/jobs.py` → `close_job` 函数（约第 161-163 行）
- 原因：代码只做了 `application.state = ApplicationState.REJECTED.value` 和 `application.outcome = "岗位关闭"`，没有调用动作引擎或手动创建 Event
- 后果：
  1. 时间线上没有"结束"记录，HR 看不到原因和时间
  2. Stage 派生不会更新（因为没有新 Event），Application 的 stage 仍停留在之前阶段（如"面试"），但 state 已是 REJECTED
  3. 简报模块的待办/日程可能仍包含这些已被 reject 的 Application
- 方案：为每个被级联 reject 的 Application 创建一条 `APPLICATION_ENDED` Event：
  ```python
  from app.models.event import Event
  from app.models.enums import EventType
  for application in active_applications:
      application.state = ApplicationState.REJECTED.value
      application.outcome = Outcome.REJECTED.value
      event = Event(
          application_id=application.id,
          type=EventType.APPLICATION_ENDED.value,
          occurred_at=now,
          actor_type="human",
          actor_id=user.id,
          payload={"outcome": "rejected", "reason": reason},
          body=f"岗位关闭：{reason}",
      )
      db.add(event)
  ```
  然后对每个 application 重新派生 stage（或直接设为 None，因为已结束）
- 影响面：岗位关闭 + 候选人时间线完整性 + 简报数据准确性

### #5 简报里"今天/明天"的面试日程，下午以后可能算错

**「简单说」**

- 什么问题：简报页面把面试分成"今天的"和"明天的"，但判断用的是英国时间（UTC），不是北京时间
- 什么时候会遇到：北京时间凌晨 0 点到早上 8 点之间查看简报（这段时间北京和英国差一天）
- 会怎样：今天上午 9 点的面试可能不显示在"今天"里；"明天"的面试可能被归到"今天"；简报顶部的"今日面试 N 场"数字不准
- 修好后：统一用北京时间判断"今天/明天"，日程分类始终准确

**「技术细节」**

- 优先级：P0
- 位置：`app/query/briefing.py` → `get_today_briefing` 函数（约第 390 行）
- 原因：代码用 `datetime.now(timezone.utc).date()` 获取"今天"的日期，但 HR 在中国（UTC+8）。UTC 的"今天"和北京时间的"今天"在 0:00-8:00（北京时间）之间是不同的日期
- 方案：用 `app/utils/time.py` 里已有的 `BIZ_TZ`（UTC+8）来计算业务日期：
  ```python
  from app.utils.time import BIZ_TZ
  now_utc = datetime.now(timezone.utc)
  today = now_utc.astimezone(BIZ_TZ).date()
  tomorrow = today + timedelta(days=1)
  ```
  同时检查 `_pending_interviews` 和 `_pending_onboards` 中的日期比较逻辑，确保 `scheduled_at` 和 `onboard_date` 也按北京时间转换后再比较
- 影响面：简报页面的日程分类 + 脉搏行数字 + 待办判断

### #6 新部署时系统设置表是空的，行为靠"碰巧"兜底

**「简单说」**

- 什么问题：就像买了一台新手机，开机后"设置"页面是空白的——没有语言、没有音量、什么默认值都没有。我们的系统第一次启动时，系统设置表里没有任何数据（"是否开放注册"、AI 配置等全是空的）
- 什么时候会遇到：全新部署环境，第一次启动系统
- 会怎样：注册功能碰巧是开放的——不是因为我们设计成"默认开放"，而是代码查设置查不到 → 变量是空的 → 判断逻辑恰好跳过了 → 注册没被拦住。这是巧合不是设计。管理员进设置页面看到空表单；想关闭注册得先手动保存一次
- 修好后：系统第一次启动时自动写入一组出厂默认值（注册开放 + 4 个空 AI 配置项），行为明确可控

**「技术细节」**

- 优先级：P1
- 位置：`app/server.py`（启动入口，第 11-13 行）
- 原因：auth contract 要求"首次启动时 system_settings 预插入 5 条记录"，但 `server.py` 只调了 `Base.metadata.create_all()` + `ensure_legacy_job_columns()` + `ensure_candidate_attachments_column()`，没有初始化 system_settings
- 后果：
  1. 注册开关靠 None 兜底：`auth.py` 查 `registration_open`，查不到时 `reg_setting` 为 None，`if reg_setting and reg_setting.value != "true"` 不成立，注册默认开放——这是隐式行为，不是显式设计
  2. GET /admin/settings 返回空对象，前端展示可能异常
  3. 关闭注册需要先手动保存一次设置
- 方案：在 `server.py` 的 `Base.metadata.create_all(bind=engine)` 之后加 seed 逻辑：
  ```python
  def ensure_system_settings():
      db = SessionLocal()
      try:
          existing = db.query(SystemSetting).count()
          if existing == 0:
              defaults = [
                  SystemSetting(key="registration_open", value="true"),
                  SystemSetting(key="ai_provider", value=""),
                  SystemSetting(key="ai_model", value=""),
                  SystemSetting(key="ai_api_key", value=""),
                  SystemSetting(key="ai_base_url", value=""),
              ]
              db.add_all(defaults)
              db.commit()
      finally:
          db.close()
  ```
- 影响面：新部署环境 + 系统管理页面 + 注册开关行为

### #7 岗位搜索框输入特殊字符时，搜索结果不对

**「简单说」**

- 什么问题：在岗位搜索框里输入 `%` 或 `_` 时，搜索结果异常。因为数据库的搜索语言里，`%` 是"万能牌"（代表任意内容），`_` 代表"任意一个字"。用户输入 `%` 本意是搜百分号，但数据库理解成"匹配一切"，于是所有岗位都出来了
- 什么时候会遇到：搜索岗位时输入了 `%` 或 `_`（不常见，但不是不可能）
- 会怎样：搜索结果不符合预期，用户困惑。注意：这不是安全漏洞，不会被黑客利用，只是搜索行为不对
- 修好后：输入什么就搜什么，特殊字符当普通文字处理。在送给数据库之前，把用户输入里的 `%` 和 `_` 前面加一个转义符号，告诉数据库"这是普通字符，不是万能牌"

**「技术细节」**

- 优先级：P1
- 位置：`app/query/jobs.py` → `list_jobs` 函数（第 27-29 行）
- 原因：代码用 f-string 拼接 LIKE 查询 `func.lower(Job.title).like(f"%{normalized_keyword}%")`，没有对 keyword 中的 SQL 通配符（`%` 和 `_`）进行转义。这不是 SQL 注入（SQLAlchemy 会参数化），但通配符未转义导致搜索语义错误
- 方案：
  ```python
  escaped = normalized_keyword.replace("%", "\\%").replace("_", "\\_")
  query = query.filter(func.lower(Job.title).like(f"%{escaped}%", escape="\\"))
  ```
- 影响面：岗位搜索功能

### #8 时间处理方式不统一，埋了一个定时炸弹 ✅ 已修复

**「简单说」**

- 什么问题：项目里生成时间戳有两种格式——一种带"这是 UTC 时间"的标签，一种不带标签。Python 规定这两种格式不能直接比大小，比了就报错崩溃。目前靠一段"创可贴"代码硬兼容着没出事
- 什么时候会遇到：目前没出问题（创可贴兜着），但未来写新代码时只要忘了贴创可贴，就会报错崩溃
- 会怎样：某个功能突然崩溃，报一个看不懂的时间类型错误
- 为什么不直接统一成北京时间：数据库里存的是"世界标准时间"（UTC），页面上展示时再自动转成北京时间。这是业界标准做法——存储用统一基准，展示按当地时区。现在的问题不是"存什么时间"，而是"生成时间戳时有的带标签有的不带"，格式不统一
- 修好后：全项目统一成"带标签的 UTC 时间"，去掉创可贴，未来写代码不用再操心格式问题

**「技术细节」**

- 优先级：P1
- 位置：多处
  - `app/utils/time.py` 的 `utc_now()`（第 10-12 行）返回 **naive** datetime：`datetime.now(timezone.utc).replace(tzinfo=None)`
  - `app/models/legacy.py` 的 `_utc_now()`（第 9-10 行）返回 **aware** datetime：`datetime.now(timezone.utc)`
  - `app/models/base.py` 的 `TimestampMixin`（第 13-21 行）用 aware datetime：`default=lambda: datetime.now(timezone.utc)`
  - `app/entry/deps.py`（第 47 行）做了兼容：`expires = token_record.expires_at.replace(tzinfo=None) if token_record.expires_at.tzinfo else token_record.expires_at`
  - 引擎层的 `_now()` 函数（如 `application_lifecycle.py`）返回 aware datetime
- 原因：历史演进——旧模型用 `_utc_now()` 返回 aware，新工具函数 `utc_now()` 返回 naive，两套并存
- 方案：统一为 aware UTC datetime：
  1. 修改 `utc_now()` 去掉 `replace(tzinfo=None)`，直接返回 `datetime.now(timezone.utc)`
  2. 去掉 `deps.py` 里的 `replace(tzinfo=None)` 兼容代码
  3. 全局搜索所有 `utc_now()` 调用点确认兼容性
- 影响面：全局时间处理，需要全面回归测试

### #9 两个 HR 同时编辑同一条记录，后保存的会悄悄覆盖前一个

**「简单说」**

- 什么问题：系统有一个防止"两个人同时改同一条数据互相覆盖"的机制。原理是给每条数据编一个版本号（第 1 版、第 2 版……），你打开编辑时记住当前版本号，保存时系统检查：如果版本号对得上就保存成功，如果对不上说明别人已经改过了，提示你刷新。但这个机制有个后门——如果请求里没带版本号，检测就直接跳过了
- 什么时候会遇到：两个 HR 同时打开同一个候选人/岗位的编辑页面，各自修改后保存
- 会怎样：后保存的人会悄悄覆盖先保存的人的修改，没有任何提示，数据丢失
- 修好后：所有写操作都强制带版本号，两个人同时编辑时系统会提示"数据已被其他人修改，请刷新后重试"

**「技术细节」**

- 优先级：P1
- 位置：`app/engine/version.py` → `check_version` 函数（第 7-13 行）
- 原因：代码 `if client_version is None: return`——注释写了"兼容旧客户端"，但意味着乐观锁保护不是强制的
- 方案：
  1. 短期：检查所有前端写操作是否都传了 version，确保无遗漏
  2. 长期：把 `client_version is None` 改为报错 `raise HTTPException(422, "缺少 version 字段")`，强制所有客户端传 version
- 影响面：所有带乐观锁的写操作（candidates / jobs / events / suppliers / terms / expenses / system_settings）

### #10 登录令牌明文存在数据库里，泄露了就能直接冒充用户

**「简单说」**

- 什么问题：用户登录后系统发的"通行证"（token）是原样存在数据库里的，就像把门禁卡密码明文写在纸上锁在抽屉里
- 什么时候会遇到：数据库文件被非法获取时（比如服务器被入侵、备份文件泄露）
- 会怎样：拿到数据库的人可以直接用里面的 token 冒充任何用户登录，不需要知道密码
- 修好后：数据库里存的是 token 的"指纹"（哈希值），即使数据库泄露，攻击者也无法用指纹反推出真正的 token。注意：这个改动会让所有现有用户需要重新登录一次

**「技术细节」**

- 优先级：P2
- 位置：`app/models/token.py`（第 16 行，token 字段直接存原始值）+ `app/entry/deps.py`（第 40 行，明文比对）+ `app/entry/api/auth.py`（第 46 行，`secrets.token_urlsafe(48)` 生成后直接存储）
- 方案：存储 token 的 SHA256 哈希值，验证时也用哈希比对：
  ```python
  import hashlib
  def _hash_token(token_str: str) -> str:
      return hashlib.sha256(token_str.encode()).hexdigest()

  # 创建时：存 _hash_token(token_str)，返回给用户原始 token_str
  # 验证时：db.query(Token).filter(Token.token == _hash_token(creds.credentials))
  ```
  这是业界标准做法（GitHub、Stripe 等都这样做）。改动会使所有现有 token 失效，所有用户需要重新登录
- 影响面：所有用户的登录安全。考虑到这是内部 HR 工具、部署在内网，风险等级中等，但修复成本很低

### #11 登录没有防暴力破解，密码可以无限试

**「简单说」**

- 什么问题：登录页面没有任何限制，可以无限次尝试密码，不会被锁定
- 什么时候会遇到：系统暴露在公网时，有人用脚本自动批量试密码
- 会怎样：攻击者可以用程序一秒试几百个密码，迟早能猜中
- 修好后：连续输错 5 次密码后，该账号锁定 15 分钟，暴力破解变得不可行

**「技术细节」**

- 优先级：P2
- 位置：`app/entry/api/auth.py` → `login` 函数（第 115-126 行）
- 原因：`login()` 只做了用户名+密码校验，没有失败计数、锁定机制或验证码
- 方案：
  1. 简单方案：在内存中记录每个 login_name 的连续失败次数，超过 5 次锁定 15 分钟
  2. 进阶方案：加 IP 级别的频率限制（如 FastAPI 的 slowapi 中间件）
  3. 对于内网部署，优先级不高，但如果未来暴露到公网则必须加
- 影响面：账号安全

### #12 「已入职」页面数据多了会变慢

**「简单说」**

- 什么问题：已入职页面加载时，系统对每条记录都单独查一次候选人信息、岗位信息、Offer 信息，记录越多查询次数越多，页面越慢
- 什么时候会遇到：已入职记录超过 20 条时开始感觉到，超过 50 条时明显变慢
- 会怎样：页面加载时间变长，100 条记录就是 300+ 次数据库查询
- 修好后：改成一次性批量查询，不管多少条记录都只查几次，页面秒开

**「技术细节」**

- 优先级：P2
- 位置：`app/query/hired.py` → `list_hired` 函数（第 38-51 行）
- 原因：经典 N+1 问题——对每个 HIRED Application 单独 `db.get(Candidate, ...)` + `db.get(Job, ...)` + 单独 query `offer_event`，100 条记录 = 300+ 次查询
- 方案：改为 joinedload 批量查询：
  ```python
  applications = (
      q.options(
          joinedload(Application.candidate),
          joinedload(Application.job),
      )
      .order_by(...)
      .offset(...)
      .limit(...)
      .all()
  )
  ```
  offer_event 也可以用 subquery 批量查询，避免逐条查
- 影响面：已入职页面性能

### #14 修改 AI 配置后，多进程部署下可能不生效

**「简单说」**

- 什么问题：管理员修改 AI 配置（比如换模型、换 API Key）后，如果系统是多进程部署的，只有当前进程会用新配置，其他进程还在用旧的
- 什么时候会遇到：用 gunicorn 多 worker 部署（如 `gunicorn -w 4`）时修改 AI 配置
- 会怎样：简历解析时好时坏——有的请求走了新配置，有的还在用旧配置，取决于请求落到哪个进程
- 修好后（长期）：改用带过期时间的缓存（比如 5 分钟自动刷新），所有进程都能及时拿到新配置
- 目前影响：Docker 部署用 uvicorn 单进程，不受影响

**「技术细节」**

- 优先级：P3（长期）
- 位置：`app/ai_client.py`（模块级 `_config` 和 `_config_loaded` 变量，第 6-7 行；`clear_config_cache()` 函数，第 46-50 行）
- 原因：`clear_config_cache()` 只修改当前进程的全局变量，其他 worker 进程的 `_config_loaded` 仍为 True，不会重新从数据库读取
- 方案：改用 TTL 缓存（如 5 分钟自动过期重读），或用 Redis 做缓存。扩展到多 worker 时再处理
- 影响面：AI 简历解析功能在多 worker 部署下的配置一致性

### #19 事件编辑功能未完全落地（菜单交互 + 编辑权限 + 控件升级 + 字段覆盖） ✅ 已修复

**「简单说」**

- 修好后：编辑按钮常显、菜单点外部关闭、所有事件可编辑、评分改为 4 档按钮、时间选择器改为 30 分钟一档、结论字段有保护

**「技术细节」**

- 优先级：P1
- 主要位置：`frontend/src/components/pipeline/EventCard.vue`（第 64-76 行菜单、第 120-136 行 `EDIT_FIELDS_MAP`、第 319-327 行 CSS `opacity: 0`）
- 设计依据：`docs/contracts/pipeline.md` 第 109 行明确规定"任意 Event 可编辑 payload/body"；`openspec/changes/archive/2026-03-10-event-edit-upgrade/design.md` 有完整设计方案

#### A. 菜单交互修复

1. **常显**：去掉 CSS `opacity: 0`，三个点按钮始终可见
2. **点外关闭**：给菜单加 click-outside 处理，点击菜单外任意位置关闭弹出菜单（现在只能再点三个点才关闭）

#### B. 编辑权限规则

事件字段分两类，编辑规则不同：

| 字段类型 | 定义 | 编辑规则 | 举例 |
|---------|------|---------|------|
| **数据字段** | 不影响流程走向的信息 | 随时可编辑 | 面试时间、面试官、Offer 薪资、入职日期、评分、备注 |
| **结论字段** | 决定了流程能否往下推进 | 如果后续已有依赖该结论的事件，必须先删除后续事件才能编辑 | 面评结论（pass/reject）、背调结果（pass/fail）、结束原因（rejected/withdrawn） |

具体的结论字段只有 3 个：
- `interview_feedback.conclusion`（面评结论）→ 后续可能有 `advance_to_offer` 等
- `background_check_result.result`（背调结果）→ 后续可能有 `offer_recorded` 等
- `application_ended.outcome`（结束原因）→ 一般是尾部事件，通常可直接编辑

实现方式：编辑结论字段时，检查该事件之后是否存在依赖事件。如有，禁用结论字段并提示"请先删除后续事件再修改此字段"。

#### C. 评分控件升级

面评评分从数字输入框改为 **4 档按钮**（不再是 1-5 分，改为 1-4 分）：

```
┌───┐  ┌───┐  ┌───┐  ┌───┐
│ 1 │  │ 2 │  │ 3 │  │ 4 │
└───┘  └───┘  └───┘  └───┘

1 = 淘汰        ← hover tooltip
2 = 通过，一般   ← hover tooltip
3 = 通过，良好   ← hover tooltip
4 = 通过，优秀   ← hover tooltip

未选中：灰色底 + 灰色数字
选中：主色底 + 白色数字（只亮一个）
悬停：浅色底 + tooltip 浮出
```

注意：评分是数据字段，修改评分不影响结论（通过/淘汰），两者独立。这个控件同时用于新建面评（FeedbackForm）和编辑面评（EventCard 编辑模式）。

### #22 面评选「淘汰→其他原因」不填原因时，按钮无反应也无提示

**「简单说」**

- 什么问题：填面试评价时，选「淘汰」→ 淘汰原因选「其他」→ 不输入具体原因 → 点「确认」，按钮没有任何反应，也没有提示"请填写原因"
- 什么时候会遇到：填面评时选了"其他"淘汰原因但忘了写具体原因
- 会怎样：用户反复点确认按钮，不知道为什么没反应
- 修好后：要么提示"请填写淘汰原因"，要么按钮显示为灰色不可点击并标注原因

**「技术细节」**

- 优先级：P2
- 位置：`frontend/src/components/pipeline/forms/FeedbackForm.vue`（第 68-70 行）
- 原因：`canSubmit` computed 正确地阻止了提交（`rejectReason.value === '__other__' && !otherRejectReason.value.trim()` 时返回 false），但没有给用户任何视觉反馈说明为什么不能提交
- 方案：在"其他原因"输入框下方加一行校验提示文字（如"请填写淘汰原因"），或者直接在按钮旁边标注
- 影响面：面评提交流程的用户体验
